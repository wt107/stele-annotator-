# stele-companion 项目深度分析报告

> 多角度分析、反向思考与改进建议

---

## 一、现状概览

### 项目基本信息
- **名称**: stele-companion (v2.0)
- **代码量**: 2,178 行 Python 代码
- **文件数**: 20 个文件（7 个源码 + 6 个测试 + 7 个文档/配置）
- **主要依赖**: python-docx, chardet, requests

---

## 二、多角度问题分析

### 🔍 角度 1: 代码质量分析

#### 发现的问题

| 问题 | 严重程度 | 具体表现 |
|-----|:-------:|:---------|
| **函数过长** | ⚠️ 中 | `build_dict()` 102 行, `annotate()` 179 行 |
| **代码分布不均** | ⚠️ 中 | `render.py` 578 行，占总数 26% |
| **异常处理不足** | 🔴 高 | 仅 10 个 try 块，覆盖率低 |
| **工具函数堆积** | ⚠️ 中 | `utils.py` 535 行，9 个函数，职责不清 |

#### 反向思考
> "如果让代码更烂会怎样？"

- 把所有代码放在一个文件 → 现在分模块是优点
- 去掉所有错误处理 → 现在至少还有基础处理
- 不写任何注释 → 现在代码有一定可读性

**结论**: 代码结构基本合理，但需要重构大函数

---

### 🔍 角度 2: 安全性分析

#### 发现的问题

| 问题 | 风险 | 位置 |
|-----|:----:|:-----|
| **文件路径注入** | 🔴 高 | 直接使用用户输入的文件路径 |
| **网络请求无限制** | ⚠️ 中 | 固定 30s 超时，无重试机制 |
| **依赖未锁定** | ⚠️ 中 | `>=0.8.10` 可能导致版本冲突 |
| **HTML 输出未转义** | 🔴 高 | 用户输入直接渲染到 HTML |

#### 反向思考
> "如何让这个项目被黑客攻击？"

1. 构造 `../../../etc/passwd` 路径读取系统文件
2. 在碑文字典中注入 `<script>alert('xss')</script>`
3. 通过超大数据包造成内存溢出

**结论**: 安全隐患明显，需要加强输入验证和输出转义

---

### 🔍 角度 3: 用户体验分析

#### 发现的问题

| 问题 | 影响 | 描述 |
|-----|:----:|:-----|
| **依赖复杂** | 🔴 高 | 需要 LibreOffice 处理 .doc |
| **网络依赖** | ⚠️ 中 | `--fetch` 功能需要联网 |
| **无图形界面** | ⚠️ 低 | 纯命令行，门槛较高 |
| **错误信息不友好** | ⚠️ 中 | 抛出异常而非友好提示 |
| **无进度显示** | ⚠️ 中 | 处理大文件时用户不知道进度 |

#### 反向思考
> "如何让用户体验更差？"

- 强制用户记住所有命令参数 → 现在有默认参数
- 处理失败不告诉原因 → 现在有错误提示
- 输出文件覆盖不警告 → 现在可能无检查

**结论**: 需要简化依赖、增加 GUI、优化错误提示

---

### 🔍 角度 4: 可维护性分析

#### 发现的问题

| 问题 | 严重程度 | 说明 |
|-----|:-------:|:-----|
| **硬编码 URL** | 🔴 高 | 4 个碑帖源 URL 写死在 utils.py |
| **无版本管理** | ⚠️ 中 | 字典格式无版本，升级困难 |
| **测试覆盖不足** | 🔴 高 | 仅 1 个集成测试文件 |
| **文档与代码分离** | ⚠️ 中 | SKILL.md 描述可能与实际代码不符 |
| **无 CI/CD** | ⚠️ 低 | 无自动化测试和发布流程 |

#### 反向思考
> "如何让项目无法维护？"

- 把所有配置写死在代码里 → 现在 URL 已经硬编码了
- 不写测试 → 现在只有基础测试
- 随意更改接口 → 现在接口相对稳定

**结论**: 需要解耦配置、增加测试、建立 CI/CD

---

### 🔍 角度 5: 扩展性分析

#### 发现的问题

| 问题 | 限制 | 描述 |
|-----|:----:|:-----|
| **固定数据源** | 🔴 高 | 只能爬取预设的 4 个碑帖源 |
| **输出格式单一** | 🔴 高 | 仅支持 HTML，不支持 PDF/图片 |
| **字典格式固定** | ⚠️ 中 | JSON 结构无法扩展新字段 |
| **无插件机制** | ⚠️ 中 | 无法自定义渲染器 |

#### 反向思考
> "如何让项目完全无法扩展？"

- 把所有功能写死 → 现在有一定模块化
- 不允许自定义样式 → 现在有 theme 参数
- 强制使用特定字体 → 现在 font-size 可调

**结论**: 需要设计插件系统、支持更多格式

## 三、核心问题清单

### 🔴 P0 - 必须修复

1. **安全隐患**: HTML 输出未转义，存在 XSS 风险
2. **路径注入**: 未验证用户输入的文件路径
3. **硬编码**: 碑帖源 URL 写死在代码中

### ⚠️ P1 - 强烈建议

4. **函数过长**: core.py 和 render.py 需要重构
5. **测试不足**: 仅 1 个测试文件，覆盖率未知
6. **依赖复杂**: LibreOffice 安装门槛高

### 💡 P2 - 优化建议

7. **无 GUI**: 考虑增加 Web 界面
8. **无进度条**: 大文件处理无反馈
9. **错误处理**: 异常信息不够友好

## 四、改进建议

### 短期（1-2 周）

1. **安全加固**
   ```python
   # 添加 HTML 转义
   from html import escape
   output = escape(user_input)
   
   # 验证文件路径
   from pathlib import Path
   input_path = Path(input_path).resolve()
   if not input_path.is_file():
       raise ValueError(f"无效文件: {input_path}")
   ```

2. **配置外置**
   ```yaml
   # config/sources.yaml
   sources:
     wikisource:
       url: "https://zh.wikisource.org/wiki/{name}"
       parser: "wikisource"
   ```

3. **增加测试**
   - 单元测试覆盖核心函数

   - 安全测试覆盖输入验证

### 中期（1 个月）

   - 拆分大函数（超过 50 行的函数拆分）
   - utils.py 按功能拆分为多个模块

5. **Web 界面**
   ```python
   # 使用 streamlit 快速构建 GUI
   import streamlit as st
   ```

6. **进度反馈**
   ```python
   from tqdm import tqdm

       process(paragraph)
   ```

### 长期（3 个月）

7. **插件系统**

   - 支持自定义字典格式

8. **更多格式**

## 五、反向思考总结

### 项目的优点（从反面看缺点不存在）

✅ **模块分离**: 没有把 2000+ 行代码写在一个文件  
✅ **基础测试**: 比完全没有测试好  
✅ **文档存在**: 有 SKILL.md 说明使用方法  
✅ **命令入口**: 提供了 pip 安装方式  
✅ **错误处理**: 有基本的 try-except  

### 项目的风险（从攻击者角度看）

⚠️ **可被 XSS 攻击**: 用户输入直接渲染 HTML  
⚠️ **可被路径遍历**: `../../etc/passwd` 可能可读  
⚠️ **依赖地狱**: LibreOffice 安装复杂  
⚠️ **维护困难**: 硬编码导致难以更新数据源
